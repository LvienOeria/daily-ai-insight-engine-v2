# AI 使用方式、Prompt 设计与错误处理

## 目的

本文档说明每日 AI 洞察引擎 MVP 中 AI/LLM 能力的使用方式。

系统将 LLM 视为受控工作流中的分析模块，而非不受限制的报告撰写者。

## AI 使用概览

LLM 用于以下场景：

1. 结构化抽取
   - 从新闻文本中抽取实体、技术、事件类型、关键事实、风险、机会和置信度。

2. 受限的中文网络搜索兼容
   - 仅对白名单中的量子位、机器之心和知乎中文网页使用 DeepSeek 网络搜索（通过 MCP 工具层），作为 API/RSS 覆盖不足时的补充。
   - 保留查询文本、返回的 URL、摘要/正文、可见日期和收集时间戳以确保可复现性。
   - 将网络搜索视为数据采集辅助手段，而非证据验证。

3. 语义事件聚类
   - 判断多条新闻是否描述同一事件。

4. 重要性解释
   - 基于结构化证据解释排名靠前的事件为何重要。

5. 日报生成
   - 将结构化事件数据转换为可读的中文分析报告。

6. 质量审查
   - 检测缺乏支撑的论断、缺失章节、薄弱逻辑和可能的幻觉。

LLM 不用于：

- 编造新闻
- 编造来源、日期、URL、公司或事实
- 替代字段验证
- 直接从未经清洗的原始数据生成报告
- 做出缺乏支撑的投资或政策论断
- 在 MVP 中文源路径中通过搜索访问非白名单的中文网页

## Prompt 设计原则

每个可复用的 prompt 或 skill 应明确：

- 角色
- 输入格式
- 输出格式
- 必填字段
- 允许的枚举值
- 证据要求
- 置信度要求
- 禁止行为
- 错误处理行为

## 核心 Prompt/Skill 模块

MVP 使用以下可复用的 skill 模块：

- `data-source-evaluation` — 数据源评估
- `news-structuring` — 新闻结构化
- `event-clustering` — 事件聚类
- `importance-scoring` — 重要性评分
- `daily-report-generation` — 日报生成
- `report-quality-check` — 报告质量检查

## 结构化抽取 Prompt 模式

关键约束：

- 输出有效的 JSON。
- 仅使用输入文本。
- 未知字段应为 `null`、`[]` 或 `other`。
- 重要判断需提供证据。
- 需包含置信度。
- 不得输出缺乏支撑的事实。

## 报告生成 Prompt 模式

关键约束：

- 仅从结构化事件数据生成报告。
- 顶级事件必须引用事件 ID 或新闻 ID。
- 趋势必须有多个事件或关键事实支撑。
- 风险与机会必须映射到结构化标签或证据。
- 不得夸大弱证据。

## 错误类型

### 数据错误

示例：

- 缺少标题
- 缺少来源
- 缺少发布时间
- 缺少摘要/正文
- 日期格式不一致
- 重复新闻
- AI 相关性弱
- 来自非白名单中文站点的搜索结果
- 搜索结果中无可追溯的可见日期

处理方式：

- 规范化日期和来源名称。
- 标记缺失字段。
- 删除或降低低质量数据的权重。
- 尽可能保留可追溯的原始载荷。
- 拒绝或隔离白名单以外的中文搜索结果。

### LLM 输出错误

示例：

- 无效 JSON
- 缺少必填字段
- 无效枚举值
- 分数超出有效范围
- 证据缺失
- 缺乏支撑的事实
- 低置信度抽取

处理方式：

- 重新运行或修复无效 JSON。
- 在适当时将无效枚举值映射为 `other`。
- 将缺失的可选字段填充为 `null` 或 `[]`。
- 删除幻觉事实。
- 将低置信度条目送审。

### 分析错误

示例：

- 不相关的条目被合并为一个事件
- 顶级事件排名缺乏证据
- 基于单条弱条目的趋势判断
- 缺乏支撑的风险或机会论断

处理方式：

- 要求事件关联新闻 ID。
- 使用基于规则的评分加上 LLM 解释。
- 要求每个主要论断都有证据支撑。
- 人工审查顶级事件。

### 输出错误

示例：

- 图表计数与结构化数据不匹配
- 报告提及不存在的新闻
- 数据源说明缺失
- Schema 说明过于浅显

处理方式：

- 运行最终质量检查。
- 验证图表数据源。
- 逐项检查所有报告论断是否与结构化数据一致。

## 人工审查点

以下情况需要人工审查：

- 最终源选择
- Schema 审批
- 低置信度的重要新闻
- Top 3-5 事件排名
- 最终报告结论
- 文档准确性

## 局限

- LLM 抽取仍可能遗漏细微上下文。
- 部分来源可能只提供简短摘要。
- 小样本量限制了趋势置信度。
- 对于高影响力的论断，人工审查仍然必要。
