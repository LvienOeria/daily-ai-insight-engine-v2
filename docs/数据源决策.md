# 数据源决策

## 目的

本文档记录每日 AI 洞察引擎的数据源选择过程。

## 源选择策略

MVP 数据采集使用三种方式：

1. **API** — 结构化、字段完整、可复现（arXiv）
2. **RSS** — 稳定的 feed，含标题、摘要、日期和 URL（TechCrunch AI、The Verge）
3. **直连 HTTP 抓取** — 用于没有 RSS 的站点（量子位）

## 必填字段

主分析条目必须包含：

- 标题
- 摘要/正文/描述
- 来源
- 发布时间（或可从 URL 推断的日期）
- URL

缺少标题、来源或发布时间的条目将被排除。

## 源分级

- **core**：稳定、字段完整，用于主数据集
- **auxiliary**：提供有用信号但可能存在字段缺失，用于上下文
- **future**：有价值但暂不可用
- **reject**：缺少关键字段或 AI 相关性低

## 中文源评估

三个中文源的实测结果：

| 源 | 方式 | 结果 | 层级 | 原因 |
|---|---|---|---|---|
| 量子位 | 直连 HTTP | ✅ 10 条，字段完整 | core | 可访问，文章页返回 200 |
| 机器之心 | 直连 HTTP | ❌ SSL 超时 | future | 非中国 IP 被 geo-block |
| 知乎 | 直连 HTTP | ❌ 需要登录 | future | 需认证 |

同时测试了 DeepSeek API 的 `enable_search` 参数作为代理，但 API 实际不触发网络搜索——该功能仅在网页聊天界面可用。

## 最终选定源

| 源 | 类型 | 方式 | 语言 | 典型条目数 | 局限 |
|---|---|---|---|---|---|
| arXiv | 研究 | API | 英文 | ~20 | 仅摘要 |
| 量子位 | 媒体 | HTTP 抓取 | 中文 | ~10 | 日期从 URL 推断；HTML 结构可能变化 |
| TechCrunch AI | 媒体 | RSS | 英文 | ~19 | AI 分类专属；3 天窗口内条目少 |
| The Verge | 媒体 | RSS + 过滤 | 英文 | ~1 | 综合 feed；需 AI 关键词过滤 |

## 已知局限

- 数据集上限 20 条（可在 `defaults.json` 中配置）
- 3 天窗口排除了更新频率较低的优质源中的条目
- 中文覆盖仅限于单一来源（量子位）
- HTTP 抓取较脆弱；站点 HTML 变化可能导致提取失败

## 实时更新的可行性

当前架构定位为**日更**系统。讨论过三种粒度的可行性：

| 粒度 | 数据采集 | 管线延迟 | 可行性 |
|:---|:---|:---|:---|
| 日更 | RSS/HTTP 按需抓取 | 2-3 分钟 | ✅ 当前模式 |
| 小时更 | RSS 缓存 TTL 调为 5 分钟 | 2-3 分钟 | ⚠️ 技术上可行但不是真实时 |
| 分钟更 | RSS 轮询 + 增量推送 | 需架构改动 | ❌ 不推荐 |

**为什么不做实时：**

1. **管线瓶颈**：每次全流程跑完需要 6 次 DeepSeek API 调用（4 次结构化 + 1 次聚类 + 1 次报告），总耗时 2-3 分钟。这是硬延迟，不是采集层的限制。

2. **爬虫不适合轮询**：HTTP 爬虫（量子位）高频抓取会被反爬，IP 封禁风险高。RSS 可以轮询但 arXiv 每天只更新一次，实时无意义。

3. **成本不可控**：分钟级运行意味着每天 1440 次全管线，DeepSeek API 费用爆炸。

4. **定位匹配**：日报的作用是提供每日 AI 行业概览，不需要秒级响应。2-3 分钟的延迟对日报来说完全可接受。

**如果想做到小时级，只需要调一行配置**：RSS 缓存 TTL 从 3600s 改为 300s。但管线延迟不变，这不是"实时"，是"更快刷新的日更"。

**真正的实时方案需要换架构**：接入新闻 API（NewsAPI/GNews）秒级响应 + 增量更新只处理新增条目 + LLM 调用改为异步流式。这是下一阶段的工程决策，MVP 不做。
